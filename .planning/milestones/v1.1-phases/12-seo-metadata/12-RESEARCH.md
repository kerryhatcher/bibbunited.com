# Phase 12: SEO & Metadata - Research

**Researched:** 2026-03-25
**Domain:** Next.js App Router Metadata API, Sitemap generation, Open Graph tags
**Confidence:** HIGH

## Summary

Phase 12 consolidates scattered, inconsistent metadata across all pages into a unified system using Next.js App Router's built-in Metadata API. The current codebase has three key problems: (1) page titles manually append `| BIBB United` even though the root layout already has a `title.template: '%s | BIBB United'` -- producing double suffixes like `"News | BIBB United | BIBB United"`, (2) no canonical URLs or consistent OG tags on static civic pages, and (3) the sitemap is generated by `next-sitemap` (a third-party postbuild tool) which has ESM/CJS compatibility issues noted in STATE.md.

The fix is straightforward: create a shared `generatePageMeta` helper in `src/lib/metadata.ts` that all pages call, replace `next-sitemap` with a native `sitemap.ts` App Router route handler that queries Payload directly, and replace the static `robots.txt` with a `robots.ts` handler. All of these are first-class Next.js features with stable APIs documented in the official docs.

**Primary recommendation:** Build the `generatePageMeta` helper first, then refactor all pages to use it, then add sitemap.ts/robots.ts, and finally clean up next-sitemap artifacts.

### Pattern 1: generatePageMeta Helper
**What:** A single shared function that takes page-specific data and returns a complete `Metadata` object with canonical URL, OG tags, Twitter card, and fallback OG image from SiteTheme.
**When to use:** Every page's `generateMetadata` export calls this instead of building metadata manually.

```typescript
// Source: Next.js official docs - generateMetadata API
// src/lib/metadata.ts

import type { Metadata } from 'next'
import { getPayload } from 'payload'
import configPromise from '@payload-config'

type PageMetaInput = {
  title: string
  description?: string
  slug?: string           // e.g., 'news' or 'news/my-article'
  type?: 'website' | 'article'
  image?: string | null   // absolute URL or relative path to media
  publishedTime?: string
  author?: string
}

const SITE_URL = 'https://www.bibbunited.com'

async function getDefaultOgImage(): Promise<string> {
  try {
    const payload = await getPayload({ config: configPromise })
    const siteTheme = await payload.findGlobal({ slug: 'site-theme' })
    const ogImage = typeof siteTheme?.ogDefaultImage === 'object'
      ? siteTheme.ogDefaultImage?.url
      : null
    if (ogImage) return `${SITE_URL}${ogImage}`
  } catch {
    // Fall through to static fallback
  }
  return `${SITE_URL}/og-default.png`
}

export async function generatePageMeta(input: PageMetaInput): Promise<Metadata> {
  const { title, description, slug, type = 'website', image, publishedTime, author } = input

  const canonicalPath = slug ? `/${slug}` : ''
  const ogImage = image
    ? (image.startsWith('http') ? image : `${SITE_URL}${image}`)
    : await getDefaultOgImage()

  return {
    title,  // Short title -- layout template appends "| BIBB United"
    description,
    alternates: {
      canonical: canonicalPath || '/',
    },
    openGraph: {
      title,
      description: description || undefined,
      url: canonicalPath || '/',
      siteName: 'BIBB United',
      type,
      images: [{ url: ogImage, width: 1200, height: 630 }],
      ...(type === 'article' && publishedTime && { publishedTime }),
      ...(type === 'article' && author && { authors: [author] }),
    },
    twitter: {
      card: 'summary_large_image',
      title,
      description: description || undefined,
      images: [ogImage],
    },
  }
}
```

**Key details from official docs:**
- `alternates.canonical` resolves against `metadataBase` set in the root layout. So passing `'/news'` as canonical produces `https://www.bibbunited.com/news`.
- `openGraph` fields from a child page **replace** (not merge with) the parent layout's `openGraph`. This means the helper must set `siteName`, `type`, and `images` every time -- they are NOT inherited from layout.
- `title` as a plain string works with the layout's `title.template`. Use `title: { absolute: '...' }` only for homepage.

### Pattern 2: Native Sitemap Handler
**What:** An async `sitemap.ts` at the app root that queries Payload for all published content.
**When to use:** Replaces next-sitemap entirely.

```typescript
// Source: https://nextjs.org/docs/app/api-reference/file-conventions/metadata/sitemap
// src/app/sitemap.ts

import type { MetadataRoute } from 'next'
import { getPayload } from 'payload'
import configPromise from '@payload-config'

const SITE_URL = 'https://www.bibbunited.com'

export default async function sitemap(): Promise<MetadataRoute.Sitemap> {
  const payload = await getPayload({ config: configPromise })

  // Fetch all published news posts
  const newsPosts = await payload.find({
    collection: 'news-posts',
    where: { _status: { equals: 'published' } },
    limit: 1000,
    select: { slug: true, updatedAt: true },
  })

  // Fetch all published CMS pages
  const pages = await payload.find({
    collection: 'pages',
    where: { _status: { equals: 'published' } },
    limit: 1000,
    select: { slug: true, updatedAt: true },
  })

  const newsEntries: MetadataRoute.Sitemap = newsPosts.docs
    .filter((p) => p.slug)
    .map((post) => ({
      url: `${SITE_URL}/news/${post.slug}`,
      lastModified: new Date(post.updatedAt),
      changeFrequency: 'weekly' as const,
      priority: 0.7,
    }))

  const pageEntries: MetadataRoute.Sitemap = pages.docs
    .filter((p) => p.slug)
    .map((page) => ({
      url: `${SITE_URL}/${page.slug}`,
      lastModified: new Date(page.updatedAt),
      changeFrequency: 'monthly' as const,
      priority: 0.8,
    }))

  return [
    {
      url: SITE_URL,
      lastModified: new Date(),
      changeFrequency: 'daily',
      priority: 1,
    },
    {
      url: `${SITE_URL}/news`,
      lastModified: new Date(),
      changeFrequency: 'daily',
      priority: 0.8,
    },
    {
      url: `${SITE_URL}/meetings`,
      lastModified: new Date(),
      changeFrequency: 'weekly',
      priority: 0.8,
    },
    {
      url: `${SITE_URL}/contact-officials`,
      lastModified: new Date(),
      changeFrequency: 'monthly',
      priority: 0.8,
    },
    ...pageEntries,
    ...newsEntries,
  ]
}
```

**Critical placement note:** `sitemap.ts` MUST be placed at `src/app/sitemap.ts` (the app root), NOT inside a route group like `(frontend)`. Route groups with parentheses do NOT create URL segments, but sitemap.ts inside them would not be recognized as the root sitemap handler.

### Anti-Patterns to Avoid

- **Manual title suffix appending:** The current code does `title: \`\${title} | BIBB United\`` in every page. This produces double suffixes because the layout already has `title.template: '%s | BIBB United'`. Pages should return short titles only.
- **Incomplete OG overrides:** When a child page sets `openGraph`, it REPLACES the parent's entire `openGraph` object (shallow merge). Always include `siteName`, `type`, and `images` in every page's OG metadata -- they do NOT inherit from layout when `openGraph` is set.
- **Relative OG image URLs without metadataBase:** OG images should work with metadataBase resolution. The root layout already sets `metadataBase: new URL('https://www.bibbunited.com')`, so relative paths like `/og-default.png` resolve correctly.
- **Sitemap inside route group:** Placing `sitemap.ts` inside `(frontend)/` would not serve it at `/sitemap.xml`. It must be at the app root.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Sitemap XML generation | Custom XML string building | `MetadataRoute.Sitemap` return type from `sitemap.ts` | Next.js handles XML serialization, headers, and caching automatically |
| Robots.txt | Static file or manual response | `MetadataRoute.Robots` return type from `robots.ts` | Keeps it in code, consistent with sitemap approach |
| Canonical URL construction | Manual URL building per page | `alternates.canonical` + `metadataBase` | Next.js resolves relative paths against metadataBase automatically |
| OG tag rendering | Manual `<meta>` tags in JSX | `openGraph` field in Metadata object | Next.js handles proper tag placement in `<head>` |

## Common Pitfalls

### Pitfall 1: Double Title Suffix
**What goes wrong:** Title renders as "News | BIBB United | BIBB United"
**Why it happens:** Page manually appends `| BIBB United` to title AND layout has `title.template: '%s | BIBB United'`. Both apply.
**How to avoid:** Pages return short titles only (e.g., `title: 'News'`). The layout template handles the suffix. Homepage uses `title: { absolute: '...' }` to bypass the template entirely.
**Warning signs:** Any page's generateMetadata returning a title string containing "BIBB United".

### Pitfall 2: OG Fields Lost on Override
**What goes wrong:** A page sets `openGraph: { title: 'My Page' }` and loses `siteName`, `type`, and `images` from the layout.
**Why it happens:** Next.js metadata merging is SHALLOW -- setting `openGraph` on a child replaces the entire parent `openGraph` object, not individual fields.
**How to avoid:** The `generatePageMeta` helper always returns a complete `openGraph` object with all required fields. Never partially override OG in page metadata.
**Warning signs:** Missing `og:site_name` or `og:image` tags when inspecting page source.

### Pitfall 3: Sitemap Caching
**What goes wrong:** New content doesn't appear in sitemap.xml after publishing.
**Why it happens:** Next.js caches `sitemap.ts` by default (it's a special Route Handler). It only regenerates on rebuild or when using dynamic APIs.
**How to avoid:** Since `sitemap.ts` calls `getPayload()` which performs a database query, it should naturally use dynamic rendering. If needed, add `export const dynamic = 'force-dynamic'` to ensure fresh data. For this site's scale, static generation at build time with ISR would also work fine.
**Warning signs:** Sitemap content unchanged after adding new articles.

### Pitfall 4: Missing OG Image for Pages Without Images
**What goes wrong:** Social media previews show a broken image or no image.
**Why it happens:** Page has no featured image and no SEO image set, and the OG metadata doesn't include a fallback.
**How to avoid:** The 4-step fallback chain (per-page SEO image -> featured image -> SiteTheme ogDefaultImage -> static /og-default.png) ensures every page always has an OG image.
**Warning signs:** Sharing a page on social media shows no preview image.

### Pitfall 5: postbuild Script Failure After Removal
**What goes wrong:** Build fails because `postbuild` script still references `next-sitemap`.
**Why it happens:** Incomplete cleanup -- removed the package but not the script entry.
**How to avoid:** Remove all four artifacts together: `next-sitemap.config.cjs`, `package.json` postbuild script, `next-sitemap` dependency, and `public/sitemap*.xml` + `public/robots.txt` static files.
**Warning signs:** CI/CD build failure after dependency removal.

### News Article Metadata (article type with published_time)

```typescript
// src/app/(frontend)/news/[slug]/page.tsx

export async function generateMetadata({ params }: Args): Promise<Metadata> {
  const { slug } = await params
  const payload = await getPayload({ config: configPromise })
  const result = await payload.find({
    collection: 'news-posts',
    where: { slug: { equals: slug } },
    depth: 1,
    limit: 1,
  })
  const post = result.docs[0]
  if (!post) return { title: 'Not Found' }

  const seoImage = typeof post.meta?.image === 'object' ? post.meta.image?.url : null
  const featuredImg = typeof post.featuredImage === 'object' ? post.featuredImage?.url : null

  return generatePageMeta({
    title: post.meta?.title || post.title,
    description: post.meta?.description || undefined,
    slug: `news/${slug}`,
    type: 'article',
    image: seoImage || featuredImg || null,
    publishedTime: post.publishDate,
    author: typeof post.author === 'object' ? post.author?.displayName : undefined,
  })
}
```

## Open Questions

1. **SiteTheme OG image caching**
   - What we know: `getPayload()` calls inside `generateMetadata` are automatically memoized within a single request per Next.js docs. Multiple pages calling `getDefaultOgImage()` during the same render cycle share one DB call.
   - What's unclear: Whether the SiteTheme fetch during sitemap generation adds meaningful overhead.
   - Recommendation: Rely on Payload's request deduplication for now. The site is small scale -- no caching layer needed. If performance becomes an issue, wrap with React `cache()`.

2. **Sitemap dynamic vs static behavior**
   - What we know: `sitemap.ts` is cached by default unless it uses dynamic APIs. Calling `getPayload()` may or may not trigger dynamic rendering depending on Next.js's detection.
   - What's unclear: Whether the Payload database call is detected as a dynamic API in Next.js 15+.
   - Recommendation: Test after implementation. If sitemap is stale, add `export const dynamic = 'force-dynamic'` or `export const revalidate = 3600` (hourly) to ensure freshness.
