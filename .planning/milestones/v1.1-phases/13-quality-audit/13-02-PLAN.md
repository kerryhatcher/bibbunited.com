**Step 3: Run audit tests (per D-05, D-06, D-08, D-09, D-10)**

Run the new audit specs. IMPORTANT: Lighthouse tests should run with limited parallelism to avoid resource contention and score variability:
```bash
npx playwright test e2e/audit/ --workers=1 --reporter=html
```

Using `--workers=1` because:
- Lighthouse is CPU-intensive; parallel runs produce unreliable scores (anti-pattern from research)
- Remote debugging port conflicts are avoided with single worker (Pitfall 2)

If Lighthouse tests fail on threshold (scores < 95):
- Run the failing test 3 times to check for variability
- If median of 3 runs is >= 95, the test is passing (document the variance)
- If median is consistently < 95, document the actual scores in AUDIT-RESULTS.md -- this is valuable diagnostic data even if the threshold is not met
- Do NOT lower thresholds -- D-05 is locked at 95+

If axe-core tests report violations:
- Document every violation with rule ID, impact level, affected elements, and route
- These are genuine accessibility issues to address
- Do NOT add exclusions to suppress violations -- D-06 is locked at zero violations
