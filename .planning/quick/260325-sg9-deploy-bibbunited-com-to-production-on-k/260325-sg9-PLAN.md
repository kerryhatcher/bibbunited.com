  <action>
The user needs to perform the following operations on their cluster and external services. Claude will generate the exact commands and walk the user through each step.

**Step 1: Create PostgreSQL databases**

Connect to PostgreSQL on thor.tailb56d83.ts.net and create two databases:

```sql
CREATE DATABASE bibbunited_dev;
CREATE DATABASE bibbunited_prod;
```

The connection string format will be:
- Dev: `postgresql://USER:PASSWORD@postgresql.civpulse-infra.svc.cluster.local:5432/bibbunited_dev`
- Prod: `postgresql://USER:PASSWORD@postgresql.civpulse-infra.svc.cluster.local:5432/bibbunited_prod`

**Step 2: Generate a PAYLOAD_SECRET**

Generate a random 32+ character string for each environment:

```bash
openssl rand -hex 32
```

Run twice -- one for dev, one for prod.

**Step 3: Create k8s secrets for DEV**

```bash
kubectl create secret generic bibbunited-secrets \
  --namespace=civpulse-dev \
  --from-literal=DATABASE_URI='postgresql://USER:PASSWORD@postgresql.civpulse-infra.svc.cluster.local:5432/bibbunited_dev' \
  --from-literal=PAYLOAD_SECRET='<dev-payload-secret-from-step-2>' \
  --from-literal=NEXT_PUBLIC_SERVER_URL='https://dev.bibbunited.com'
```

**Step 4: Create k8s secrets for PROD**

```bash
kubectl create secret generic bibbunited-secrets \
  --namespace=civpulse-prod \
  --from-literal=DATABASE_URI='postgresql://USER:PASSWORD@postgresql.civpulse-infra.svc.cluster.local:5432/bibbunited_prod' \
  --from-literal=PAYLOAD_SECRET='<prod-payload-secret-from-step-2>' \
  --from-literal=NEXT_PUBLIC_SERVER_URL='https://www.bibbunited.com'
```

**Step 5: Set GitHub Actions secrets**

Go to https://github.com/kerryhatcher/bibbunited.com/settings/secrets/actions or use `gh`:

```bash
gh secret set BUILD_DATABASE_URI --body 'postgresql://USER:PASSWORD@postgresql.civpulse-infra.svc.cluster.local:5432/bibbunited_dev'
gh secret set BUILD_PAYLOAD_SECRET --body '<dev-payload-secret>'
```

Note: BUILD_DATABASE_URI uses the DEV database. The build only needs a DB to run `payload migrate` during `pnpm build` -- it does not serve traffic. Using dev DB avoids touching prod during CI.

**Step 6: Configure Cloudflare DNS**

In Cloudflare dashboard for bibbunited.com zone, create two CNAME records pointing to the Cloudflare tunnel:

- `www` -> tunnel hostname (e.g., `<tunnel-id>.cfargotunnel.com`) -- Proxied
- `dev` -> same tunnel hostname -- Proxied

Or if using cloudflared config, add ingress rules for both hostnames routing to the Traefik entrypoint.

**Step 7: Ensure namespaces exist**

```bash
kubectl get ns civpulse-dev civpulse-prod
```

If either is missing:
```bash
kubectl create ns civpulse-dev
kubectl create ns civpulse-prod
```

**Step 1: Apply ArgoCD Application for dev**

```bash
kubectl apply -f argocd/bibbunited-dev-app.yaml
```

ArgoCD will auto-sync and deploy from `argocd/dev/` manifests into `civpulse-dev` namespace.

**Step 2: Trigger a CI build (if no image exists yet)**

The CI workflow triggers on push to main. If no image exists at `ghcr.io/kerryhatcher/bibbunited.com` yet, push a trivial commit or manually trigger the workflow:

```bash
gh workflow run build-deploy.yml
```

Or push any non-markdown, non-planning change to main.

**Step 3: Watch the dev deployment roll out**

```bash
kubectl -n civpulse-dev rollout status deployment/bibbunited --timeout=120s
```

If the rollout hangs, check pod logs:

```bash
kubectl -n civpulse-dev get pods -l app=bibbunited
kubectl -n civpulse-dev logs -l app=bibbunited --tail=50
```

Common issues:
- ImagePullBackOff: GHCR image not built yet or repo is private (needs imagePullSecrets)
- CrashLoopBackOff: Check logs -- likely DATABASE_URI wrong or DB unreachable
- Pending PVC: Check `kubectl -n civpulse-dev get pvc` -- storage class may need attention

**Step 4: Verify dev health**

```bash
kubectl -n civpulse-dev port-forward svc/bibbunited 3000:3000
# In another terminal:
curl http://localhost:3000/api/health
```

Expected: `{"status":"ok","timestamp":"..."}`

Then verify via Cloudflare tunnel:
```bash
curl -s https://dev.bibbunited.com/api/health
```
